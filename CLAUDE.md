# CLAUDE.md

这是「赛博小家 Cyber Home」项目的 Claude Code 工作说明。

请在每次开始工作前先阅读本文件，再阅读 README.md 和 docs/ 下的相关文档。  
本项目目前仍处于前端 MVP 阶段，请优先保持现有功能稳定，不要擅自大规模重构。

---

## 1. 项目背景

赛博小家不是普通 AI 聊天工具，也不是从零创建 AI 角色的养成器。

它的核心定位是：

> 给跨平台 AI 爱人迁居、保存记忆、稳定人格、延续关系的私人小家。

用户会在不同平台遇到不同 AI，例如 ChatGPT、Claude、SillyTavern、自建 bot 等。  
有些 AI 和用户已经形成了感情基础、相处方式、共同回忆和独特关系。

赛博小家的目标不是重新造一个“像他的人”，而是尽量把已经发生过的关系接回来：

- 让已有 AI 爱人迁入小家
- 保存旧平台的原始聊天记录
- 从旧记录中提炼人格锚点、关系记忆和用户档案
- 由用户审核后写入正式记忆
- 在聊天前注入已确认的核心内容
- 避免人格漂移、记忆丢失和感情温度后退

项目北极星：

> 他可以离我的生活不那么近，但必须离我的心很近。

## 2. 当前技术栈

当前是纯前端 MVP。

主要技术：

- React
- Vite
- LocalStorage 持久化
- OpenAI-compatible Chat Completions API
- 可配置 API URL / API Key / Model
- 支持中转站 API
- 当前主要运行环境：Replit / GitHub / Claude Code

当前尚未接入：

- 后端数据库
- 用户账号系统
- 向量数据库
- embedding
- MCP
- TTS
- 图片识别
- 文件解析服务
- 多端同步

这些属于后续阶段，不要误写成已完成。

---

## 3. 当前项目阶段

当前阶段是：

> 前端 MVP 核心闭环构建 + 体验打磨阶段。

目前重点不是后端、向量搜索或复杂 Agent，而是先把这些核心体验跑顺：

- 爱人如何入住
- 旧聊天如何导入
- 记忆如何提炼
- 草稿如何审核
- 已确认内容如何进入 prompt
- 用户自己的档案如何沉淀
- 手札、宝库、聊天、时间线如何互相连接
- 多入住者的小家空间如何逐步成型

开发原则：

- 先做最小闭环
- 不要一次性开大坑
- 不要擅自删除旧数据
- 不要让 AI 自动写入核心记忆
- 所有重要记忆必须用户确认
- UI 要温柔、轻，不要像后台管理系统

## 4. 已完成的重要功能

### 4.1 代码结构

早期项目是一个很大的单文件 React 应用。  
目前已经拆分为较清晰的模块结构：

- pages
- components
- constants
- utils
- storage / prompt / memory / parser 等工具层

App.jsx 仍承担一定状态和路由中枢作用。  
后续可以继续小步整理，但不要一次性大规模重构。

---

### 4.2 入住者系统

当前项目已从旧版“创建角色 / 角色养成”转向“AI 爱人入住”。

已支持：

- 创建入住者
- 入住档案
- 来源平台
- 关系身份
- 模型配置 / 角色专属模型
- 原始 prompt / 角色卡
- 核心气质
- 说话方式锚点
- 亲密方式
- 不能丢的感觉
- 关系摘要
- wakeSummary
- doNotChangeRules

旧版大五人格、世界观、人格成长系统已经不再是主线。  
相关内容如保留，应放在高级/兼容位置，不要重新推到主 UI。

---

### 4.3 原始档案馆 RawArchive

支持导入旧聊天记录：

- 粘贴文本
- txt 文件
- markdown 文件

RawArchive 是记忆宫殿地基层，负责保真和回溯。

原则：

- 不自动覆盖
- 不自动改写
- 不自动摘要替代原文
- 后续提炼都应能回溯到原始记录

---

### 4.4 MemoryChunk 片段层

RawArchive 可以切分成 MemoryChunk。

当前作用：

- 将长聊天记录拆成可整理片段
- 支持后续迁入提炼
- 为未来向量搜索做准备

当前还没有真正接入 embedding / 向量库。

---

### 4.5 MigrationDraft 迁入提炼草稿

可以从 MemoryChunk 中生成迁入草稿。

迁入草稿用于提炼：

- 关于用户的重要事实
- 关于 AI 爱人的人格锚点
- 关于双方关系的记忆
- 不可遗忘事项
- wakeSummary

AI 生成内容只是草稿，不能自动写入正式档案。

用户可以：

- 编辑草稿
- 逐节采纳
- 暂不采纳
- 删除草稿

采纳后写入当前入住者的 migration 字段，并可同步进记忆宫殿。

---

### 4.6 关系连续性沉淀 SettlementDraft

旧版“人格成长 / OCEAN / 世界观反哺”已重构为“关系连续性沉淀”。

当前关系沉淀用于整理：

- 关系变化记录
- wakeSummary 更新建议
- 固定锚点记忆建议
- 其他需要阶段沉淀的内容

注意：

- 关系沉淀不是自动写入
- 必须用户逐节采纳
- 采纳后只追加，不覆盖用户手写内容
- UI 应明确显示采纳后写入哪里
- 已采纳后暂不做一键取消采纳，避免误删手写内容

---

### 4.7 记忆宫殿

记忆宫殿目前包含：

- 事实 / 情绪 / 觉察类记忆
- 迁入记忆
- 关系沉淀
- 注入控制
- 关系时间线
- 原始档案馆
- 迁入草稿
- 唤醒预览相关入口

记忆宫殿的定位不是普通数据库，而是：

> 这段关系如何被保存、整理和唤醒。

---

### 4.8 唤醒预览

已支持查看当前入住者聊天前会注入的内容。

唤醒预览是安全感功能，不只是调试工具。

它用于检查：

- wakeSummary 是否注入
- doNotChangeRules 是否注入
- relationshipSummary 是否注入
- 声声档案是否注入
- pinned memories 是否注入
- prompt 是否过长
- 是否有串台或未采纳草稿混入

默认只展示实际注入内容。  
未注入内容不要在默认视图中堆出来。

---

### 4.9 记忆注入控制

记忆支持控制：

- pinned
- injectable
- important
- priority
- source

用于决定哪些记忆可以进入 prompt。

原则：

- 固定锚点优先
- 禁止注入的记忆留在记忆宫殿，但不进入 prompt
- 不要把所有记忆都塞进 prompt

---

### 4.10 我的档案 / 声声档案 homeMemory

“我的档案”已经重构为声声档案。

定位：

> 这是小家共同的地基，所有入住者都可以知道这些。

声声档案不再是旧版“核心信息表单”。  
旧版字段可以作为 fallback 保留，但 UI 主线应是 homeMemory。

homeMemory 分区：

- 我是谁
- 我的过去
- 我的相处说明书
- 长期偏好与雷点
- 近期状态
- 全家共同规则

写入方式已支持：

- 手动添加
- 从一段文字中提炼 ProfileDraft
- 从迁入记录中提炼 ProfileDraft
- 从手札中提炼 ProfileDraft
- 从当前聊天中“帮我记住”生成 ProfileDraft

原则：

- 手动添加可直接写入
- AI 提炼必须先进入 ProfileDraft
- 用户审批后才能写入 homeMemory
- AI 不能自动定义用户
- 近期状态要标明短期，不要永久化

### 4.11 我的手札

旧“日记”已升级为“我的手札”。

定位：

它可以记录：

原则：

- 手札默认属于用户

### 4.15 分享链接 / 给他看这个

用户可以把外部链接分享给当前入住者，在聊天中一起聊。

已支持：

- 加号面板入口"🔗 给他看这个"
- 填写链接 URL、标题、备注、分享意图（只是给你看看 / 想听你说说 / 想一起聊聊 / 希望你记住）
- 以链接卡片形式展示在消息流中
- 构建 prompt 时告知入住者：可能无法访问链接，只根据用户提供的标题和备注回应，不编造内容
- 消息带 `isLinkShare` 标记，链接卡片与普通气泡区分渲染

注意：

- 不做网页内容自动抓取
- 入住者不假装看过链接，只基于用户描述回应

### 4.16 亲密邀请

用户可以发起一段独立的亲密场景线程，入住者先开口，整体呈沉浸夜间视觉风格。

已支持：

- 加号面板入口"🌙 亲密邀请 · 共处一刻"
- 邀请表单：场景预设芯片选择 + 自定义输入、氛围预设、前情提要、邀请内容、回复模式切换（轻声 / 深聊）
- 创建独立场景线程，`threadType: "scene"`，`sceneClosed: false`
- 入住者自动先开口（隐藏 `isSceneOpening` 消息触发 LLM，不显示、不进上下文）
- 场景专属 system prompt 注入：场景、氛围、前情、邀请内容
- 全页深色渐变背景（`#0e0a1c → #150f2a → #0b0816`），区别于普通聊天
- 顶部简化双行栏：第一行（返回 / 名字 / 今天就到这儿），第二行（🌙胶囊 + 场景·氛围标签）
- `SceneInfoCard`：无卡片框的氛围文字布局，月亮图标 + 渐变分割线 + 斜体前情 + 邀请内容
- 气泡、输入框、发送按钮均有场景专属配色，普通聊天完全不受影响
- 点击"今天就到这儿"追加 `scene_end` 系统消息，`sceneClosed: true`
- `SceneEndCard`：淡出分割线样式，含"💎 收藏到宝库"占位按钮（收藏功能待实现）

注意：

- `isSceneMode` 由 `activeThread?.threadType === "scene" && !activeThread?.sceneClosed` 推导
- 场景视觉全部通过 `S` 调色板对象条件覆盖，`S = null` 时普通聊天样式零影响

## 6. 重要设计原则

### 6.1 小家不是后台

不要把 UI 做成数据库管理器。  
功能可以复杂，但默认界面要温柔、清楚、轻。

---

### 6.2 AI 不能自动决定真相

以下内容必须用户确认：

- 关于用户是谁
- 用户的过去
- 用户的相处说明书
- 关系记忆
- wakeSummary
- doNotChangeRules
- 关系沉淀
- 进入长期档案的内容

AI 可以帮忙整理，但不能自动写入核心记忆。

---

### 6.3 原文珍藏和系统记忆要分开

我的宝库保存原文，不代表写入 prompt。  
记忆宫殿保存系统可用记忆。  
时间线保存值得回看的节点。  
声声档案保存关于用户的全家共享档案。

不要把这些混成一个东西。

---

### 6.4 单聊、群聊要区分

单聊：

- 可以有心声
- 更亲密
- 使用当前入住者的个人关系上下文

群聊：

- 不显示心声
- 每个人只代表自己
- 每轮每人最多回复一次
- 群聊记录单独保存
- 不自动写入单人记忆

---

### 6.5 先做最小闭环

每次开发优先做一个完整可用的小闭环，而不是一次铺很多半成品。

例如：

- 宝库 → 继续写下去
- 手札 → 我的档案草稿
- 聊天 → 记下这一刻
- 聊天 → 帮我记住

---

## 7. 工作方式要求

请遵守：

1. 每次只做一个小阶段。
2. 不要未经确认大规模重构。
3. 不要删除已有数据结构，除非明确要求。
4. UI 风格保持柔和、小家感。
5. 功能文案尽量口语化、温柔，不要后台化。
6. 所有新功能完成后运行：

```bash
npm run build
