# 他的房间 · 设计文档

> 状态：设计草案，中期空间化重构方向，暂不实现。  
> 第一版最小方案可在现有基础上增量实现。

---

## 功能定位

每个入住者在小家里都有自己的房间。

现在的问题是：与某个入住者相关的所有功能（聊天、档案、记忆宫殿、时间线、唤醒预览…）散落在不同页面入口里，彼此割裂。用户需要在多个页面之间来回跳转，没有一个「属于 ta 的地方」的感觉。

「他的房间」的目标是：给每个入住者一个聚合空间，所有和 ta 有关的功能都从这里进入。

这是小家**空间化**的第一步——先有房间，再慢慢填入家具。

---

## 房间里有什么

以下功能都属于「他的房间」的组成部分：

| 功能 | 当前状态 | 房间集成方式 |
|------|----------|-------------|
| 聊天 | 已实现 | 主入口，从房间直接进入 |
| 入住档案（人格、关系、规则等） | 已实现 | 房间里的「档案柜」 |
| 记忆宫殿（事实/情绪/觉察/沉淀） | 已实现 | 房间里的「记忆墙」 |
| 关系时间线 | 已实现 | 房间里的「回忆相框」 |
| 唤醒预览 | 已实现 | 房间里的「今日状态」 |
| 入住仪式 / 纪念 | 已实现（仪式消息） | 房间角落的纪念位置 |
| 迁入记录 / 原始档案 | 已实现 | 房间里的「旧物箱」 |
| 他的宝库（未实现） | 未实现 | 房间里的「珍藏盒」 |
| 我们的照片回忆（未实现） | 未实现 | 房间里的「相册墙」 |
| 人格锚点 / 世界观 | 已实现（在档案里） | 房间里的「人格板」 |
| 形象 / 头像 | 已实现（emoji/图片） | 房间中央 |

## 和现有页面的关系

当前的功能分布（现状）：

```
入口页
  └─ 卧室
       ├─ 手札
       ├─ 宝库
       ├─ 便签墙
       └─ 门 → 聊天页（含顶栏跳转到档案、记忆宫殿）
  └─ 成员档案列表
       └─ 档案编辑 → 记忆宫殿 / 时间线 / 唤醒预览 / 迁入记录
```

目标状态（中期）：

```
入口页
  └─ 卧室
       ├─ 手札
       ├─ 宝库
       ├─ 便签墙
       └─ 门 → [他的房间]
                 ├─ 聊天
                 ├─ 记忆宫殿
                 ├─ 关系时间线
                 ├─ 他的宝库
                 ├─ 我们的照片回忆
                 ├─ 唤醒预览
                 └─ 入住档案编辑
```

**档案编辑页和记忆宫殿页不消失**，只是入口从「成员档案列表」逐步迁移到「他的房间」。

## 第一版最小方案

不做视觉重构，只做一个**入住者聚合入口页**（`CharRoomPage`）。

### 功能

- 展示入住者头像、名字、关系身份
- 下方列出该入住者相关的所有功能入口：
  - 开始聊天（主按钮）
  - 记忆宫殿
  - 关系时间线
  - 唤醒预览
  - 入住档案
  - 迁入记录 / 原始档案

### 入口

- 从卧室「门」选择入住者后，先进入「他的房间」，再从房间里点「开始聊天」
- 或者：门直接进聊天，另开一个「房间」入口（如成员头像长按 / 档案列表里的房间按钮）

### 触发时机

从卧室「门」点击某个入住者后：
- 当前逻辑：直接进入聊天页
- 新逻辑（最小版）：先进入该入住者的聚合入口页，由用户决定下一步

这个改动影响 `enterChat` 的调用时机，需要谨慎处理。

## 后续方案：逐步收进房间

分阶段进行，不一次性大改：

**阶段 1**（最小版，可先做）：聚合入口页，功能列表。  
**阶段 2**：加入简要摘要展示——最近一条记忆、最近时间线事件、唤醒摘要前两句。  
**阶段 3**：空间化视觉设计，房间有背景图、家具热区（桌子=档案，书架=记忆，窗户=时间线）。  
**阶段 4**：「他的宝库」和「照片回忆」落地并集成进房间。  
**阶段 5**：形象系统——入住者的立绘/头像在房间里展示。

## 注意事项

- 「他的房间」是**入住者维度**的页面，不是全局页面
- 不同入住者的房间互相独立，不共享数据
- 现有的成员档案列表（`ProfilesPage`）继续存在，作为管理视图
- 记忆宫殿、时间线等页面逻辑不变，只是入口路径调整
- `enterChat(charId)` 的行为不变，聚合页只是在其前面增加一步
